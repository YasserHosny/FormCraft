# Implementation Plan: Recurring Subscription Billing with Proration and Self-Serve Downgrade

**Branch**: `059-recurring-billing` | **Date**: 2026-06-20 | **Spec**: [spec.md](spec.md)

---

## Summary

Extends the F058 PayGateway billing system to make tier subscriptions recurring (monthly or annual auto-renewal via Stripe Subscriptions). Adds server-side prorated upgrade charges, self-serve downgrade scheduling at renewal, self-serve cancellation, and automated dunning that downgrades orgs after a configurable number of failed renewal payments. Core security invariant from F058 is preserved: all amounts are computed server-side; the client never supplies a monetary value.

## Phase 0: Research Summary

All research decisions are documented in [research.md](research.md). Key resolutions:

- **Webhook verification**: Inherits F058's HMAC-SHA256 via `verify_webhook_signature()`. New subscription webhook endpoint uses identical mechanism.
- **Proration strategy**: FormCraft computes preview amount for UX display; actual prorated charge processed by Stripe via `proration_behavior=create_prorations` on the upgrade call; confirmed via `invoice.paid` webhook.
- **Subscription webhook endpoint**: Separate `POST /api/billing/paygateway/subscription-webhook` (not merged into existing payment webhook — different schema, different handler).
- **Customer Portal URL**: PayGateway exposes `POST /api/v1/customers/{id}/portal`; FormCraft backend proxies and returns short-lived URL; Angular opens in new tab.
- **Interval switching**: Rejected (409) with `billing.interval_change_not_supported` — out of scope for 059.
- **Dunning threshold config**: Read from `org_settings` key `billing.dunning_max_failures`; platform default = 3.
- **Idempotency**: `last_invoice_id` column on `billing_subscriptions` prevents duplicate processing of the same Stripe invoice event.

## Implementation Notes

### Subscription Creation Flow
1. Client calls `POST /api/billing/subscriptions` with tier + interval.
2. Backend checks no active/past_due subscription exists.
3. Backend lazily creates Stripe Customer via PayGateway if `stripe_customer_id` is null on org.
4. Backend looks up `billing_prices.provider_price_id` for the tier+interval combination.
5. Backend calls PayGateway `POST /api/v1/subscriptions`.
6. Stripe creates a subscription; the first invoice may require 3DS (`requires_action`). If so, return `checkout.client_token` to Angular for `confirmCardPayment()` (same as F058 flow).
7. On `invoice.paid` webhook: update `billing_subscriptions.status='active'`, create `billing_fulfillments` row.

### Proration Preview (Upgrade)
1. Client calls `POST /api/billing/subscriptions/upgrade` with new tier.
2. Backend computes: `preview = (new_amount × days_left/period_days) − (old_amount × days_left/period_days)`.
3. Backend calls PayGateway `POST /api/v1/subscriptions/{id}/upgrade` with new `price_id` and `proration_behavior=create_prorations`.
4. Stripe generates a prorated invoice immediately; `invoice.paid` webhook fires when charged.
5. Backend confirms upgrade on webhook — updates subscription tier.

### Dunning Flow
1. `invoice.payment_failed` received → verify `invoice_id != last_invoice_id` (idempotency).
2. Increment `failed_payment_count`. Set `status='past_due'`. Update `last_invoice_id`.
3. Dispatch email via `NotificationService`.
4. If `failed_payment_count >= threshold` (from `org_settings`): set `status='cancelled'`, revert org to Starter, write `billing_fulfillments` row with `source='webhook'`, write audit log.

### Scheduled Downgrade on Renewal
1. `invoice.paid` received for a subscription with `scheduled_downgrade_tier` set.
2. Instead of keeping current tier: update org's `subscription_tier` to `scheduled_downgrade_tier`, clear `scheduled_downgrade_tier`, update subscription `tier` field.
3. Write `billing_fulfillments` row.
