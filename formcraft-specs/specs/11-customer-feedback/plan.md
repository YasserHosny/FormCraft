# Implementation Plan: Customer Feedback

**Branch**: `001-customer-feedback` | **Date**: 2026-05-07 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `formcraft-specs/specs/11-customer-feedback/spec.md`

## Summary

Allow authenticated FormCraft users to submit feedback (text + optional image + optional audio) via a persistent floating widget on all pages. Audio input supports both live browser recording and audio file upload. Submitted feedback is stored in a new `feedback_submissions` table in Supabase. Admins review feedback through a dedicated `/admin/feedback` dashboard with status management.

## Technical Context

**Language/Version**: Python 3.12 (backend), TypeScript / Angular 17 (frontend)  
**Primary Dependencies**: FastAPI, Supabase (PostgreSQL + Storage + Auth), Angular Material  
**Storage**: Supabase Storage (`feedback` bucket) for images and audio; PostgreSQL for submission records  
**Testing**: pytest (backend unit + integration), Angular TestBed (frontend)  
**Target Platform**: Web (server: Linux, client: modern browsers)  
**Project Type**: Web application — backend API + Angular SPA  
**Performance Goals**: Widget interactive within 1 second; file upload feedback immediate (client-side validation before upload)  
**Constraints**: 30-second submission cooldown per user (server-enforced); max 5 MB images, 10 MB audio; 2000 char text limit; 2-minute recording cap  
**Scale/Scope**: Single-tenant; all authenticated roles can submit; only ADMIN role can read

## Implementation Phases

### Phase 1 — Backend Foundation (DB + Upload + Submit)

1. Write migration `008_create_feedback_submissions.sql`
2. Write `app/schemas/feedback.py` (Pydantic models for all request/response shapes)
3. Write `app/services/feedback/service.py`:
   - `upload_file(user_id, file, bucket_path)` → storage path
   - `delete_file(user_id, storage_path)` → removes orphaned storage object on client-side abort
   - `submit_feedback(user_id, payload)` → checks cooldown, inserts row
   - `list_feedback(filters, page, limit)` → paginated query; JOINs `profiles` on `user_id` to resolve `submitter_display_name` (display_name, fallback to email); returns signed URLs for image/audio
   - `update_status(id, status)` → patch row
4. Write `app/api/routes/feedback.py` (submit + upload + delete-upload endpoints)
5. Extend `app/api/routes/admin.py` (list + status-update endpoints)
6. Register `feedback.router` in `app/main.py`

### Phase 2 — Backend Tests (TDD)

Write all test files before implementation passes:

- `tests/unit/feedback/test_feedback_service.py`:
  - Cooldown blocks submission within 30 s
  - Cooldown allows submission after 30 s
  - Empty text rejected
  - Text > 2000 chars rejected
  - Image URL stored when provided
  - Audio URL stored when provided
  - Null image/audio accepted
- `tests/integration/feedback/test_feedback_route.py`:
  - `POST /api/feedback` → 201 valid submission
  - `POST /api/feedback` → 429 cooldown active
  - `POST /api/feedback/upload/image` → 200 valid image
  - `POST /api/feedback/upload/image` → 413 oversized
  - `POST /api/feedback/upload/image` → 400 bad MIME
  - `POST /api/feedback` → 401 unauthenticated
- `tests/integration/feedback/test_admin_feedback_route.py`:
  - `GET /api/admin/feedback` → 200 paginated list (admin)
  - `GET /api/admin/feedback` → 403 non-admin
  - `PATCH /api/admin/feedback/{id}` → 200 status updated
  - `PATCH /api/admin/feedback/{id}` → 400 invalid status
  - `PATCH /api/admin/feedback/{id}` → 404 not found

### Phase 3 — Frontend Widget

1. Create `feedback.module.ts` and `feedback.service.ts`
2. Build `FeedbackWidgetComponent`:
   - Floating action button (bottom-right, all authenticated pages)
   - Modal/overlay with: text area (2000 char limit + counter), image picker (JPEG/PNG/WEBP, 5 MB, thumbnail preview), audio section (record button via MediaRecorder API + file upload input), submit/cancel buttons, success/error states
   - 300 ms debounce on submit to prevent double-click
   - Client-side file validation before upload attempt
   - Cooldown timer feedback (show remaining seconds on 429)
3. Build `FeedbackAdminComponent`:
   - Table with columns: date, user, page, text (truncated), image (thumbnail), audio (playback), status chip
   - Status filter (new / reviewed / resolved)
   - Click row to expand full text
   - Status dropdown per row to update
4. Wire routes and mount widget in `app.component.html`
5. Add i18n keys to `ar.json` and `en.json`

### Phase 4 — Integration & Polish

1. Apply migration (`supabase db push`)
2. Create `feedback` storage bucket
3. Manual end-to-end test: submit text-only, submit with image, submit with recorded audio, submit with uploaded audio file, check admin dashboard, change status
4. Verify cooldown behaviour (two rapid submissions)
5. Verify graceful audio degradation in a browser without MediaRecorder
