# Implementation Plan: Feedback Threading & Replies

**Branch**: `014-feedback-threading` | **Date**: 2026-05-09 | **Spec**: [spec.md](spec.md)  
**Depends on**: Feature 011 (`001-customer-feedback`) — `feedback_submissions` table, admin dashboard, and submission widget must exist

## Summary

Add two-way threaded replies to FormCraft feedback submissions. Admins reply from the dashboard; submitters read and reply on a new `/my-feedback` page. In-app notifications are delivered via Supabase Realtime for active sessions and queued for offline users. The admin unread badge clears automatically when the thread is expanded.

## Technical Context

**Language/Version**: Python 3.12 (backend), TypeScript / Angular 17 (frontend)  
**Primary Dependencies**: FastAPI, Supabase (PostgreSQL + Realtime), Angular Material  
**Testing**: pytest (backend unit + integration), Angular TestBed (frontend)  
**Target Platform**: Web (server: Linux, client: modern browsers — Chrome, Firefox, Safari, Edge)  
**Performance Goals**: Reply visible in admin thread within 1 s (Realtime push — no reload); in-app notification delivered within 5 s; `/my-feedback` loads within 2 s for 500 submissions  
**Constraints**: Max 2000 chars per reply; thread shows 20 most-recent by default (cursor pagination via `before_id`); no editing or deletion of replies; no `read_at` on individual reply rows  
**Scale/Scope**: Single-tenant; reply endpoints available to all authenticated users (scoped to own submissions) and admins

## Implementation Phases

### Phase 1 — Backend: Migration, Schemas, Services, Routes

1. Write `011_create_feedback_replies.sql` — full SQL from data-model.md:
   - `ALTER TABLE feedback_submissions ADD COLUMN reply_count`, `has_unread_user_reply`
   - CREATE `feedback_replies` with index + RLS
   - CREATE `feedback_notifications` with indexes + RLS

2. Write `app/schemas/reply.py`:
   - `ReplyCreateRequest(text_content: str)` — `@field_validator`: 1–2000 chars
   - `ReplyResponse(id, author_role, author_name, text_content, created_at)`
   - `ThreadResponse(replies: list[ReplyResponse], has_earlier: bool)`
   - `MyFeedbackItem(id, page_url, text_content, status, submitted_at, reply_count, has_unread_admin_reply)`
   - `MyFeedbackResponse(results: list[MyFeedbackItem], total, page, page_size)`
   - `NotificationResponse(id, feedback_id, reply_id, created_at, delivered_at, read_at)`
   - `NotificationsResponse(notifications: list[NotificationResponse], unread_count)`

3. Write `app/services/feedback/reply_service.py`:
   - `get_replies(feedback_id, user_id, is_admin, limit, before_id)` → validates ownership for non-admins; applies cursor; returns `ThreadResponse`
   - `post_reply(feedback_id, author_id, author_role, payload)` → INSERT reply; UPDATE `reply_count += 1`; if `author_role == 'user'`: UPDATE `has_unread_user_reply = TRUE`; if `author_role == 'admin'`: INSERT notification for submission's `user_id`; return `ReplyResponse`
   - `mark_submission_read(feedback_id)` → UPDATE `has_unread_user_reply = FALSE`
   - `get_my_feedback(user_id, page, page_size)` → SELECT submissions owned by user; LEFT JOIN `feedback_notifications` to compute `has_unread_admin_reply` (any unread notification for this submission); return `MyFeedbackResponse`
   - `get_notifications(user_id)` → SELECT all notifications for user; UPDATE `delivered_at = NOW()` on undelivered rows atomically; return `NotificationsResponse`
   - `mark_notification_read(notification_id, user_id)` → UPDATE `read_at = NOW()`; 403 if not owner; 404 if not found

4. Update `app/api/routes/feedback.py`:
   - `GET /api/my-feedback` → `get_my_feedback()`
   - `GET /api/feedback/{id}/replies` → `get_replies()` with ownership check
   - `POST /api/feedback/{id}/replies` → `post_reply(author_role='user')`
   - `GET /api/notifications` → `get_notifications()`
   - `PATCH /api/notifications/{id}/read` → `mark_notification_read()`

5. Update `app/api/routes/admin.py`:
   - `GET /api/admin/feedback/{id}/replies` → `get_replies()` (admin — no ownership check)
   - `POST /api/admin/feedback/{id}/replies` → `post_reply(author_role='admin')`
   - `PATCH /api/admin/feedback/{id}/read` → `mark_submission_read()`

### Phase 3 — Frontend

1. Create `FeedbackRealtimeService` in `formcraft-frontend/src/app/features/feedback/services/feedback-realtime.service.ts`:
   - Singleton Angular service (`providedIn: 'root'`)
   - `subscribeToThread(feedbackId: string): Observable<ReplyResponse>` — creates `thread:{feedbackId}` Supabase channel; emits on `INSERT` to `feedback_replies` for this `feedback_id`; returns `Subject` that completes on `unsubscribeThread(feedbackId)`
   - `unsubscribeThread(feedbackId: string)` — removes the channel
   - `notificationEvents$: Observable<NotificationResponse>` — global channel; subscribes to `INSERT` on `feedback_notifications WHERE recipient_user_id = auth.uid()` once on service init; broadcasts to any subscriber
   - `destroy()` — removes all channels (called on logout)

2. Create `ThreadComponent` in `formcraft-frontend/src/app/shared/components/thread/`:
   - `@Input() replies: ReplyResponse[]`
   - `@Input() hasEarlier: boolean`
   - `@Input() loading: boolean`
   - `@Output() loadEarlier = new EventEmitter<string>()` — emits `before_id` of oldest visible reply
   - `@Output() sendReply = new EventEmitter<string>()` — emits `text_content`
   - Template: `*ngFor` reply list (author chip, timestamp, text); "Load earlier messages" button (shown when `hasEarlier`); reply textarea + Send button; character counter (2000 limit); empty-thread message
   - Shared between admin dashboard expanded row and `/my-feedback` expanded submission

3. Build `MyFeedbackModule` (`my-feedback.module.ts`):
   - `MyFeedbackComponent`: calls `GET /api/my-feedback` on init; `mat-expansion-panel` per submission; on panel open → call `GET /api/feedback/{id}/replies`, subscribe to `FeedbackRealtimeService.subscribeToThread(id)`, append live replies; on panel close → `unsubscribeThread(id)`; calls `POST /api/feedback/{id}/replies` on Send; shows `has_unread_admin_reply` badge per item
   - `MyFeedbackService`: HTTP calls for `/api/my-feedback`, `/api/feedback/{id}/replies`, `POST /api/feedback/{id}/replies`, `/api/notifications`, `PATCH /api/notifications/{id}/read`

4. Update `FeedbackAdminComponent`:
   - Thread panel in expanded row: on row expand → call `GET /api/admin/feedback/{id}/replies` + subscribe to `FeedbackRealtimeService.subscribeToThread(id)` + call `PATCH /api/admin/feedback/{id}/read`; on row collapse → `unsubscribeThread(id)`
   - Reply input: `POST /api/admin/feedback/{id}/replies` on Send using `ThreadComponent`
   - Unread badge: show `mat-badge` on submission rows where `has_unread_user_reply === true`; badge cleared reactively when `PATCH /read` succeeds

5. Add notification badge to shell/nav:
   - On app init (post-auth): call `GET /api/notifications` → seed `unreadCount`
   - Subscribe to `FeedbackRealtimeService.notificationEvents$` → increment `unreadCount` on each new notification
   - Badge navigates to `/my-feedback` on click; calls `PATCH /api/notifications/{id}/read` on notification click

6. Update `app-routing.module.ts`:
   - Add route `{ path: 'my-feedback', loadChildren: () => import('./features/my-feedback/my-feedback.module').then(m => m.MyFeedbackModule), canActivate: [AuthGuard] }`

7. Add i18n keys to `ar.json` + `en.json`:
   - `feedback.thread.reply`, `feedback.thread.send`, `feedback.thread.load_earlier`, `feedback.thread.empty`, `feedback.thread.char_count`
   - `feedback.thread.admin_label`, `feedback.thread.user_label`
   - `my_feedback.title`, `my_feedback.empty_state`, `my_feedback.submission_date`, `my_feedback.status`
   - `notifications.badge_title`, `notifications.mark_read`
